# Dimension Tokens

> Semantic dimension tokens тАФ every length-based value in the system, covering both sizing (the element itself, mainly `ui_scale`) and spacing (gaps between or around elements, mainly `density` and `typography_context`). Surface in CSS with px or rem units preserved.
>
> *(`ob.s.dimension.token_family_docs.$description` тАФ [`03_semantic/dimension/density/static.json`](../../../../src/lib/themes/03_semantic/dimension/density/static.json))*

## At a glance

| Layer | Path | Source files | Purpose |
|---|---|---|---|
| **Primitive** | `ob.p.dimension.{px\|rem}.*` | [`02_primitive/dimension.json`](../../../../src/lib/themes/02_primitive/dimension.json) | Atomic length values. Not for direct consumption. |
| **Semantic** | `ob.s.dimension.{static\|dynamic}.*` | [`03_semantic/dimension/{group}/{mode}.json`](../../../../src/lib/themes/03_semantic/dimension/) | Named tokens organised by 3 mode groups. |
| **Component** | (per-component) | various | Consumes semantic dimension tokens. |

The live visual reference is the Figma page **`ЁЯУР Dimension тАУ Spacing & Sizing YYYY-MM-DD HH:MM`** in `DesignSystem@Tokens V9.7`, generated by [`scripts-custom/figma-doc-builders/dimension/dimension.js`](../../../../scripts-custom/figma-doc-builders/dimension/dimension.js).

## Primitive layer

| Scale | Path | Count | Keying |
|---|---|---|---|
| Absolute px | `ob.p.dimension.px.<N>` | 28 | Key `<N>` is the px count itself (`8` тЖТ `8px`). |
| Fluid rem | `ob.p.dimension.rem.<N>` | 26 | Key `<N>` is a t-shirt index `0тАУ2500`; the `$description` records the px equivalent (`1rem тЖТ "16px equivalent"`). |

The two scales are **parallel**: every primitive px has a corresponding rem at the same magnitude (computed from a 16 px root).

- **`px`** тАФ absolute, does not scale with the user's browser font-size setting. Use for elements that must render pixel-perfect: borders, fine lines, icons.
- **`rem`** тАФ fluid, relative to the root `<html>` font size. Required for accessibility тАФ scales with user preferences.

**The choice between px and rem happens at the semantic layer**, not in the primitive layer itself.

> Primitives are atomic and context-agnostic. They must not be consumed directly by components.

## Semantic layer: 3 groups ├Ч static + dynamic

Semantic dimension tokens are split into three orthogonal **groups**, each with a `static` flavour (constant across modes) and 1+ dynamic mode files.

| Group | Static file | Dynamic modes | Primary use |
|---|---|---|---|
| `density` | `density/static.json` | `compact.json`, `standard.json`, `spacious.json` | **spacing** тАФ between elements |
| `ui_scale` | `ui_scale/static.json` | `sm.json`, `md.json`, `lg.json` | **sizing** тАФ of elements themselves |
| `typography_context` | `typography_context/static.json` | `interface.json`, `prose.json` | **spacing** тАФ vertical padding around typography |

### Static vs dynamic

- **Static** tokens point to a primitive directly: `"$value": "{ob.p.dimension.px.4}"`. They do not change when the mode switches.
- **Dynamic** tokens belong to one of the mode files for their group. Switching the active mode re-resolves the same token name to a different primitive (Style Dictionary "last file wins" mode resolution).

Both flavours emit a plain CSS custom property at the consumption site тАФ the dynamic switch happens entirely at build / load time.

## `density` тАФ spacing between elements

> Density-independent spacing tokens тАФ fixed values not affected by density mode. Use: for spacing that must remain constant regardless of density. Avoid: for spacing that should compress in compact or expand in spacious mode.
>
> *(`$description` from `density/static.json`)*

- **Static**: 30 tokens тАФ 15 sizes ├Ч `{px, rem}`. Sizes follow a single t-shirt + numeric scale:

  | Bucket | Sizes |
  |---|---|
  | Base scale | `xs, sm, md, lg, xl` |
  | Extended scale | `2xl, 3xl, 4xl, тАж , 11xl` |

- **Dynamic** (3 modes): same 15-size scale; each token name resolves to a different primitive depending on the active density mode:
  - `compact.json` тАФ tighter spacing
  - `standard.json` тАФ balanced spacing for general-purpose layouts
  - `spacious.json` тАФ generous spacing

See [`02-modes/05-density.md`](../02-modes/05-density.md) for mode mechanics.

## `ui_scale` тАФ sizing of elements

> Fixed component dimensions (UI Scale) тАФ values that remain constant across all size modes. Use: for dimensions that must not scale between size modes. Avoid: for dimensions that should respond to UI Scale changes.
>
> *(`$description` from `ui_scale/static.json`)*

Organised into **7 magnitude buckets**, ordered by px range (data-derived):

| Bucket | px range | Sizes present | Purpose |
|---|---|---|---|
| `none` (top-level) | 0 | (single value ├Ч `px`/`rem`) | Zero / collapse / reset. |
| `micro` | 1тАУ3 | `xs sm md` | Hairlines, sub-pixel rules, fine borders. |
| `element` | 4тАУ12 | `xs sm md lg xl` | Individual element dimensions. |
| `spacing` | 16тАУ32 | `xs sm md lg xl` | Inline-element widths. |
| `container` | 36тАУ64 | `xs sm md lg xl` | Container sizing. |
| `layout` | 80тАУ192 | `xs sm md lg xl` | Page-level layout sizing. |
| `macro` | 256+ | `xs sm` | Large-scale dimensions. |

Each `bucket ├Ч size` pair has both `px` and `rem` variants.

- **Static**: fixed values that do not respond to UI-scale mode changes.
- **Dynamic** (3 modes): values typically derived from primitives ├Ч multiplier:
  - `sm.json` тАФ Small UI scale
  - `md.json` тАФ Medium UI scale (default)
  - `lg.json` тАФ Large UI scale

See [`02-modes/03-ui-scale.md`](../02-modes/03-ui-scale.md) for mode mechanics.

## `typography_context` тАФ vertical padding around typography

> Fixed vertical spacing for HTML typography elements тАФ padding-top and padding-bottom values for headings and paragraphs. Static because this surrounding spacing remains constant regardless of typography context mode (interface vs prose).
>
> *(`$description` from `typography_context/static.json`)*

Sizes: 8 steps (`none, xs, sm, md, lg, xl, 2xl, 3xl`). **No `px`/`rem` split** тАФ each token is single-valued; `$type` is `spacing` (Tokens Studio unofficial type that compiles to `dimension` on export).

Tokens are consumed as padding on specific selectors (`h1тАУh6`, `p`, `p_lead`). Each token's `$description` records which selector consumes it тАФ e.g.:

- `xs` тАФ minimal vertical spacing. Consumed as bottom padding on all headings (`h1тАУh6`).
- `sm` тАФ small vertical spacing. Consumed as top padding on subheadings (`h2тАУh6`) and bottom padding on paragraphs.
- `2xl` тАФ generous vertical spacing. Consumed as bottom padding on lead paragraphs (`p_lead`).

- **Static**: fixed across the typography-context mode switch.
- **Dynamic** (2 modes):
  - `interface.json` тАФ compact spacing tuned for UI controls and labels.
  - `prose.json` тАФ generous spacing tuned for long-form reading.

See [`02-modes/04-typography-context.md`](../02-modes/04-typography-context.md) for mode mechanics.

## Picking px vs rem at the semantic layer

The choice is encoded in the token reference itself:

- `ob.s.dimension.static.density.md.px` тЖТ `{ob.p.dimension.px.8}` (always 8 px)
- `ob.s.dimension.static.density.md.rem` тЖТ `{ob.p.dimension.rem.600}` (always 0.5 rem тЙб 8 px at root font-size 16)

Pick the variant that matches the surface:

| Use | Pick |
|---|---|
| Body text, content padding, scalable layouts, anything that should respect user font-size preferences | **`.rem`** |
| Borders, dividers, icons, anything pixel-perfect | **`.px`** |

`typography_context` is the exception тАФ it ships only one variant (single `$value` per size, no `.px`/`.rem` suffix).

## Naming conventions

```
ob.p.dimension.<unit>.<key>                                 primitive
ob.s.dimension.<flavour>.<group>.[<bucket>.]<size>[.<unit>] semantic
```

- `<flavour>` тИИ `static | dynamic`
- `<group>` тИИ `density | ui_scale | typography_context`
- `<bucket>` only present in `ui_scale` (one of the 7 buckets above)
- `<size>` тИИ `none, xs, sm, md, lg, xl, 2xl, тАж , 11xl` (subset depends on group)
- `<unit>` тИИ `px | rem` тАФ present in `density` and `ui_scale`; absent in `typography_context`

The Figma reference page sorts tokens within each table by **size magnitude** (`none тЖТ micro тЖТ element тЖТ spacing тЖТ container тЖТ layout тЖТ macro` for ui_scale; `xs тЖТ sm тЖТ тАж тЖТ 11xl` for density), then by unit (`px` before `rem`), so a designer reads the table top-to-bottom from smallest to largest.
